# 当前 WG VPN 状况

## 适用场景

用户想查看当前 WireGuard VPN 是否在线、是否握手、peer 是否存在、服务端是否运行时，使用本模块。

## 固定入口

并行调用：

1. `openclaw_get_wg_status`
2. `clawwrt_list_devices`

职责绑定：

1. VPS 服务端状态（是否已安装、是否运行）只能基于 `openclaw_get_wg_status` 的返回结果判断。
2. 路由器客户端状态（接口/peer/握手/流量）只能基于 `clawwrt_get_wireguard_vpn_status` 的返回结果判断。
3. 未调用对应专用 API 前，不允许输出“已安装/未安装/运行正常/客户端配置正常”等结论。

## 执行流程

1. 先展示服务端状态结果。
2. 若在线设备数为 0：停止，并提示用户当前没有可查询的在线路由器。
3. 若用户未指定设备：
   1. 展示在线设备清单。
   2. 询问用户要查看哪些设备的 WG 状态。
   3. 未确认前不继续。
4. 对用户确认的每台设备，顺序调用 `clawwrt_get_wireguard_vpn_status`。
5. 输出时明确区分：
   1. 服务端状态
   2. 路由器侧接口/peer/握手状态
   3. 若有异常，指出异常位于哪一侧

## 判定规则（必须按返回结果）

1. `openclaw_get_wg_status` 返回可用时：
   1. 仅依据其返回字段判断“WG 是否已安装/服务是否运行”。
   2. 若返回显示未安装：结论只能是“当前 VPS 尚未安装 WG”。
   3. 若返回显示已安装但服务未运行：结论只能是“当前 VPS 已安装 WG，但服务未运行或状态异常”。
   4. 若返回显示已安装且服务运行：结论只能是“当前 VPS WG 服务端处于运行状态”。
2. `clawwrt_get_wireguard_vpn_status` 返回可用时：
   1. 仅依据接口状态、peer 信息、最近握手、收发流量字段判断客户端是否配置正常。
   2. 若存在 peer 且有最近握手或流量：明确写“客户端隧道已有握手/有流量，配置基本生效”。
   3. 若无 peer、无握手且无流量：明确写“客户端配置未生效或未连通”，不推断具体根因。
3. 任一 API 返回不可用时：
   1. 只能报告“对应侧状态检查失败或不可用”。
   2. 不允许跨侧反推另一侧一定正常或一定故障。

## 输出要求

1. 不把“当前 Agent 运行环境的服务端检查结果”擅自表述成“VPS 一定正常”。
2. 若服务端结果不可用，只能说“当前服务端检查失败或不可用”，不能推断 VPN 一定故障。
3. 若路由器返回有握手时间、收发流量，应明确写出“隧道已有握手/有流量”。
