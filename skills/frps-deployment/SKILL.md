# 内网穿透服务端部署指南

在 **OpenClaw VPS 宿主机**上部署并运行内网穿透服务端（代号：nwct-server），以便为连接的 龙虾WiFi 路由器实现内网穿透。

## ⚠️ 全局强规则（优先级最高，任何阶段均适用）

> 违反以下规则会导致配置流程混乱，必须严格遵守：
>
> 1. **禁止在服务端部署完成之前询问客户端（路由器侧）信息**。路由器、映射服务、远程端口等问题必须等阶段 1 全部完成后才能提出。
> 2. **Token 必须始终配置，不得留空或跳过**。Token 是服务端与客户端互信的唯一凭证，缺少 Token 会导致客户端无法连接服务端。Agent 须自动生成随机字符串作为 Token，绝对不能询问用户"是否需要我生成"，也不能以任何理由省略此步骤。用户主动提供 Token 时才使用用户的值。
> 3. **监听端口、Token、VPS IP 均来自工具返回值**，不得在部署前向用户询问这些参数；使用工具默认值即可，部署完成后再将结果告知用户。
> 4. **`server_addr` 必须由用户明确提供**，绝对不能自行猜测或用本地 IP 填充。
> 5. **每个阶段完成前必须向用户说明接下来将执行的验证步骤**，不能在没有告知的情况下直接结束会话。
> 6. **禁止在任何面向用户的输出中使用技术代号**：不得出现 `FRPS`、`frps`、`xfrpc`、`frpc` 等词。统一使用：服务端 → **内网穿透服务端**，客户端 → **内网穿透客户端**，token → **认证密钥**。工具调用参数内部使用原始字段名不受此限制。
> 7. **凡是需要用户输入的参数，必须逐项引导用户输入，并附带一句简要说明**（用途 + 填写建议），不能只抛出参数名让用户自行猜测。

## 标准提问模板（固定顺序，建议直接复用）

当进入需要用户输入参数的步骤时，优先使用以下模板逐项提问，不要改写成过于简短的参数清单。

### 模板 A：阶段 1 结束后的首轮收集

请按顺序向用户确认以下信息：

1. 请提供**目标路由器 device_id**（从设备列表中选择要下发配置的路由器）。
2. 请提供**VPS 公网 IP 或域名（server_addr）**（内网穿透客户端会连接到这个地址，不能填写内网地址或猜测值）。
3. 请说明要映射的**本地服务类型**（例如 SSH、Web、数据库；用于后续确认本地 IP、本地端口、远端端口）。

### 模板 B：创建单条映射前的参数确认

请按顺序向用户确认以下信息：

1. 请提供**本地 IP 地址**（被穿透服务实际所在主机地址，例如 `192.168.1.10`；不要填写 `127.0.0.1`，除非服务确实运行在路由器本机）。
2. 请提供**本地端口**（该服务真实监听端口，例如 SSH 常见 `22`）。
3. 请提供**远端端口**（VPS 对外开放端口，外部访问 `VPS_IP:远端端口`；需同步放行防火墙规则）。

### 模板 C：缺失 server_addr 时

若 `server_addr` 尚未给出，先单独询问：

- 请提供 **VPS 公网 IP 或域名（server_addr）**，用于让内网穿透客户端连接服务端。

## 推荐工作流 (端到端)

### ⚡ 工作流入口（收到任何内网穿透请求后的第一步）

❌ **严禁**：收到请求后先询问用户任何问题、先列出路由器设备、先询问服务器信息。

✅ **必须**：**立即调用 `openclaw_get_frps_status`** 检查服务端当前状态，根据返回结果决定后续路径：

| 返回状态 | 下一步 |
|----------|--------|
| 未安装 / 未运行 | → 直接进入**阶段 1**，自动部署，不询问任何参数 |
| 已运行，Token 为空 | → 直接进入**阶段 1**，重新部署并注入随机 Token |
| 已运行，Token 正常 | → 将端口、Token 告知用户，询问 `server_addr` 后进入**阶段 2** |
| 路由器已有客户端配置 | → **忽略旧配置，仍须先执行上表检查**；服务端状态确认正常后，以服务端返回的 Token 和端口**覆盖**路由器旧配置，再进入**阶段 2** |

❌ **严禁**：发现路由器已有旧客户端配置，就跳过服务端检查直接启用客户端。旧配置的 Token/端口可能与服务端不匹配，必须以服务端为准重新写入。

---

### 阶段 0：环境重置 (可选)

如果用户要求重置、清理或卸载之前的配置，请调用 `openclaw_reset_frps`。该工具将停止服务、清理配置及二进制文件。

---

### 阶段 1：服务端部署 (VPS 侧)

> **进入本阶段的正确行为**：检测到内网穿透服务端未安装/未运行后，**立即**执行部署，不询问任何参数。

❌ **严禁**：在调用 `openclaw_deploy_frps` 之前向用户询问任何参数（包括端口、token、服务名等）。检测到服务未安装/未运行 → 直接进入步骤 1，无需用户确认。

1. **自动化部署**：
   - 直接调用 `openclaw_deploy_frps`，使用工具内置默认值（端口 7070，自动生成 token）。
   - **参数来源说明**：端口使用工具默认值 7070；Token 由 Agent 在调用时生成随机字符串后传入工具，二者均无需用户介入，也不存在需要用户决策的场景。
   - 该工具会自动获取最新版服务端核心、安装至 `/usr/bin/nwct-server` 并配置 `nwct-server` systemd 服务。
   - ❌ **禁止**：在此步骤前向用户询问端口、token 或其他参数。

2. **状态验证**：
   - 调用 `openclaw_get_frps_status` 确认服务运行状态，获取实际的监听端口及 Token。
   - ❌ **强制检查**：若返回的 Token 为空或未设置，**必须立即重新调用** `openclaw_deploy_frps` 并显式传入随机生成的 Token，确保服务端 Token 非空后再继续。**绝对不能在 Token 为空的情况下进入阶段 2。**

3. **信息告知（强制，不可跳过）**：
   - ❌ **严禁**：在告知以下信息之前就询问用户路由器或映射服务。
   - 将以下信息**一次性**告知用户：
     - 服务端监听端口（来自 `openclaw_get_frps_status` 返回值）
     - 认证 Token（来自 `openclaw_get_frps_status` 返回值）
     - VPS 公网 IP（提示用户确认）
   - 告知用户防火墙需要开放上述端口（TCP）。
    - **然后**调用 `clawwrt_list_devices` 获取当前在线设备列表，以简短列表形式展示给用户（仅显示设备名称和 device_id），并按“**模板 A**”逐项引导用户输入。

---

### 阶段 2：客户端配置 (路由器侧)

> **进入本阶段的前提**：阶段 1 已完成，已获得监听端口、Token，且用户已明确提供 VPS 公网 IP。

1. **收集必要信息**（若阶段 1 末尾尚未收集）：
   - ❌ **禁止只列参数名**。必须逐项提问并附简要说明，建议按“**模板 A** + 模板 B”顺序收集。
   - 若需自行组织语言，至少覆盖以下要点：
     - **目标路由器 device_id**：请从设备列表中提供目标设备 ID（用于把客户端配置下发到正确路由器）。
     - **VPS 公网 IP 或域名（server_addr）**：请提供服务端公网地址（客户端将主动连接该地址，必须用户明确给出）。
     - **本地 IP 地址**：请提供被穿透服务所在主机在路由器内网中的地址（例如 `192.168.1.10`，不要填写 `127.0.0.1`，除非服务确实运行在路由器本机）。
     - **本地端口**：请提供该服务真实监听端口（例如 SSH 常见 `22`，Web 常见 `80/443`）。
     - **远端端口**：请提供要在 VPS 对外开放的端口（外部用户访问 `VPS_IP:远端端口`，并确保防火墙已放行该端口）。

2. **连接设置**：
   - 调用 `clawwrt_set_xfrpc_common`，填入阶段 1 获取的 `server_port` 和 `token`，以及用户提供的 `server_addr`。
   - 若用户尚未提供 `server_addr`，必须先按“**模板 C**”引导用户输入，获得明确值后再继续。
   - ❌ **强制校验**：调用前必须确认 `token` 字段非空。若 `token` 为空，**禁止继续执行**，须返回阶段 1 重新获取有效 Token 后再执行本步骤。

3. **服务添加**：
   - ❌ **严禁**：自行猜测或假设本地 IP 地址、本地端口、远端端口。**必须**在调用工具前向用户逐一明确确认以下三项参数：
     - **本地 IP 地址**（即路由器内网中被穿透服务所在主机的 IP，不得填写 `127.0.0.1` 或任何猜测值）
     - **本地端口**（被穿透服务实际监听的端口号）
     - **远端端口**（VPS 侧对外暴露的端口号，需与阶段 3 防火墙开放端口一致）
    - 建议直接使用“**模板 B**”提问，避免遗漏参数说明。
   - 用户明确提供上述三项参数后，再调用 `clawwrt_add_xfrpc_tcp_service` 创建映射。

4. **客户端运行检查**：
   - **必须检查** 内网穿透客户端是否正常启动并成功连接，通过 `clawwrt_get_xfrpc_config` 或系统日志确认。
   - 向用户说明将进入阶段 3 验证。

---

### 阶段 3：最终功能验证

> **向用户明确声明**："现在将在 VPS 侧验证端口是否已进入监听状态，以确认穿透链路建立成功。"

1. **端口监听验证**：
   - 在 VPS 宿主机上检查对应的远程端口（如 TCP 6000）是否已进入 `LISTEN` 状态。
   - 若端口**未监听**：
     1. 调用 `clawwrt_restart_xfrpc` 重启目标路由器的内网穿透客户端服务。
     2. 等待 1 秒后，再次在 VPS 宿主机上检查同一端口是否进入 `LISTEN` 状态。
     3. 重启后仍未监听：告知用户内网穿透客户端连接失败，引导检查认证密钥、服务器地址、防火墙配置。

2. **连通性测试**：
   - 提示用户通过 `VPS_IP:远程端口` 测试连接（如 `ssh -p 6000 user@VPS_IP`）。
   - 确认测试结果后，汇总告知用户完整的穿透配置信息。

---

> **⚠️ 防火墙提醒**：务必引导用户在 VPS 侧开放服务器端监听端口（TCP）及所有远程映射端口（TCP/UDP），否则阶段 3 验证必定失败。
